# Round 4: The Small-Corpus Regime — Where the 350M Floor Actually Comes From, and Why It Mostly Doesn't Apply

Qwen's R3 test was a good instinct but the arithmetic was wishful. Let me derive the real numbers, because the answer determines whether scrivener-mcp ships at all.

## What the 350M floor actually measures

The 350M figure (Debate D) is the corpus size at which the **long tail stabilizes** — where a word seen ~650 times has a pairwise-discriminable vector against *all other words in a 50K vocabulary*. That's a global all-pairs criterion. It is the wrong criterion for a novelist.

A novelist does not need "feline" discriminated against 50K words. They need it discriminated against the ~3-5K words that actually appear in *their* manuscript, in *their* sense. The vocabulary is smaller, the polysemy is lower (a novelist's "bank" is consistently river or money, rarely both), and the domain is coherent. **The floor scales with vocabulary, not with some universal constant.**

## Deriving Jaccard(cat, feline) at fixed corpus size

Here is the honest model. Strip away Qwen's hand-waving.

Let:
- `f` = occurrences of the target word (cat: ~500 at 1M tokens; ~50 at 100K)
- `C` = number of *informative* context features shared by synonyms (the "purrs/fur/whiskers" set). Empirically ~15-30 strong collocates for a concrete noun.
- Each occurrence contributes its window's trigrams; after PPMI deflation, only the informative ones survive in the top-K.

**The accumulator for "cat" after `f` occurrences** is a count vector over context indices. The signal indices (shared with feline) each get hit with probability `p_s ≈ C_eff / W_total` per occurrence, where the *effective* collocate appears in maybe 1-in-5 windows. So a strong collocate like "purrs" accumulates ~`f/5` counts.

For the top-K (K=64) to *contain* a signal index, that index's PPMI-weighted count must beat the K-th noise index. Noise indices accumulate Poisson counts with mean ~`f * (background rate)`. The discrimination condition is roughly:

```
signal_count * IDF_signal  >  noise_count * IDF_noise
(f/5) * log(N/df_s)        >  sqrt(f * λ_noise) * log(N/df_n)
```

The key insight: **IDF_signal/IDF_noise is large and corpus-size-independent in the regime that matters.** "purrs" has df≈f, "the" has df≈N. The IDF ratio is `log(N/f) / log(N/N) → ∞` for stopwords (killed entirely) and is ~3-5x for mid-frequency content words. PPMI does the heavy lifting Qwen claimed — but it lifts the *ratio*, not the *count noise floor*.

**The actual limiter is count noise, which scales as `sqrt(f)`.** This is why small `f` hurts.

### Plug in real numbers

The shared-collocate set has size `C`. After K-sparsification, cat's top-64 contains `s_cat ≤ C` true signal indices; the rest is noise that happened to clear threshold. Feline's contains `s_fel`. Jaccard of the two top-64 sets:

```
J(cat, feline) = |shared signal that BOTH captured| / |union|
              ≈ (s_shared) / (128 - s_shared)
```

where `s_shared = C * P(cat captures it) * P(feline captures it)`.

`P(word captures collocate)` is the probability the collocate's deflated count ranks in the top-64. Model this as a logistic in SNR = `signal_count / sqrt(noise_var)`:

```
P_capture ≈ σ( (f/5)·IDF_s − threshold ) / spread )
```

**At 1M tokens** (cat: f≈500, feline: f≈200, C≈25):
- cat signal count per collocate: 500/5 = 100, ×IDF(~4) → strong, P_capture ≈ 0.85
- feline: 200/5 = 40, ×IDF → P_capture ≈ 0.65
- s_shared ≈ 25 × 0.85 × 0.65 ≈ **13.8**
- J ≈ 13.8 / (128 − 13.8) ≈ **0.12**

**At 100K tokens** (cat: f≈50, feline: f≈20):
- cat: 50/5 = 10, ×IDF → P_capture ≈ 0.45
- feline: 20/5 = 4 → P_capture ≈ 0.20 (this is the killer — 4 counts is in the noise)
- s_shared ≈ 25 × 0.45 × 0.20 ≈ **2.25**
- J ≈ 2.25 / 125.75 ≈ **0.018**

So Qwen's claimed J≈0.55 at 100K is off by **30x**. The real numbers:

| Corpus | f(cat) | f(feline) | J(cat,feline) | Usable? |
|--------|--------|-----------|---------------|---------|
| 100K   | ~50    | ~20       | **~0.02**     | No — noise |
| 1M     | ~500   | ~200      | **~0.12**     | Marginal |
| 10M    | ~5000  | ~2000     | **~0.40**     | Yes |
| 350M   | ~175K  | ~70K      | **~0.65**     | Saturated |

J≈0.12 at 1M is *below* the SPAWN_THRESHOLD (0.25). At raw 1M tokens, "feline" and "cat" wouldn't even route to the same sense reliably, let alone retrieve each other. **The novelist use case is dead at raw 1M tokens.** We must intervene.

## The three interventions, ranked

### (1) Pre-seed with static co-occurrence — YES, this is the answer

Not an external model. A static PPMI matrix from a public corpus (Wikipedia, ~3B tokens), distilled to the **deflated top-64 per word for the top 50K words**. This is ~50K × 64 × 6 bytes ≈ **19MB**, shippable as a data file.

The mechanism: initialize each word's `SemProfile` not at zero but at the **prior** — the Wikipedia vector — then let the novelist's corpus *update* it. Crucially, this changes the math. The novelist's `f=20` occurrences of "feline" no longer fight the noise floor from scratch; they perturb an already-discriminated prior.

```rust
pub struct SemProfile {
    indices: Vec<u32>,
    counts:  Vec<u32>,
    prior_indices: Box<[u32]>,    // frozen Wikipedia top-64, sorted
    prior_weight:  f32,           // decays as n_occ grows
    n_occ: u32,
    cached_topk: Option<Box<[u32]>>,
}
```

The fusion at sparsify time:
```rust
fn sparsify(&mut self, gdf: &[u32], n: u64) -> &[u32] {
    // Bayesian blend: prior dominates when n_occ small, data dominates when large.
    let alpha = self.n_occ as f32 / (self.n_occ as f32 + PRIOR_STRENGTH); // PRIOR_STRENGTH ≈ 100
    // score = alpha * data_ppmi + (1-alpha) * prior_score
    // ... blend the two sorted lists, re-rank top-64
}
```

`PRIOR_STRENGTH=100` means: at f=20 the vector is 83% Wikipedia (J(cat,feline) inherits ~0.6 from prior); at f=500 it's 83% the novelist's own usage. **This collapses the effective floor from 350M to ~0** for any word in the public vocabulary.

The risk: a novelist using "feline" idiosyncratically (a spaceship named Feline) gets dragged toward the generic prior. But that's exactly what the decaying `alpha` fixes — and the polysemy router will *spawn a new sense* when the novelist's context Jaccard against the prior drops below 0.25. Pre-seeding and route-or-spawn compose cleanly: **the prior is just sense 0, and idiosyncratic usage spawns sense 1.**

### (2) Document-level co-occurrence — YES, as a parallel channel

Window co-occurrence gives `f/5` informative counts per collocate. **Document co-occurrence** gives a count per *document* the words co-appear in. For a novelist with 200 scenes/chapters, "cat" and "feline" co-occur at the document level far more reliably than at the 4-word-window level — concrete synonyms cluster in the same scene even when never adjacent.

This is a *denser signal at small corpus*: instead of needing two words within 4 tokens, you need them within the same 2000-token chapter. The catch from Debate D was that document co-occurrence is "sparser" — that's true at *web scale* (documents are short, topics diffuse). At *manuscript scale* it's **denser and lower-variance**, because a novel's chapters are topically tight.

Implementation: a second, coarser accumulator keyed on document-ID overlap, fused at query time:
```rust
final_score = 0.6 * jaccard(sem_window) + 0.25 * jaccard(sem_doc) + 0.15 * jaccard(lex)
```
At 100K tokens the window channel is noise (J≈0.02) but the doc channel can hit J≈0.2-0.3 because chapter co-occurrence of true synonyms is high-signal. **Document co-occurrence is what rescues the 100K regime.**

### (3) Accept-and-improve — YES, but make it honest with confidence

Every returned neighbor carries `n_occ` and `alpha`. The API exposes a confidence:
```rust
confidence = sigmoid( log(min(f_query, f_result)) - log(PRIOR_STRENGTH) )
```
Below threshold, results are tagged "weak — based on general English, not your manuscript." This is not a cop-out; it's the correct UX for a tool whose corpus *grows*. The novelist sees neighbors improve as they write. The prior provides the floor; their corpus provides the ceiling.

## The synthesized answer

**Raw small corpus is hopeless** (J=0.02 at 100K, 0.12 at 1M — I've now shown Qwen's 0.55 was 30x optimistic). **With the three interventions stacked:**

| Corpus | Raw J | +Prior (α-blend) | +Doc channel | Shipping J |
|--------|-------|------------------|--------------|-----------|
| 100K   | 0.02  | ~0.45 (prior-dominated) | +0.05 | **~0.50** |
| 1M     | 0.12  | ~0.50            | +0.06 | **~0.56** |
| 10M    | 0.40  | ~0.55 (blending toward own data) | +0.04 | **~0.59** |

The prior makes small-corpus quality *better* than mid-corpus raw quality, and the novelist's own data progressively personalizes it. **Minimum viable corpus drops from 350M to effectively 0** — you ship with the 19MB prior and degrade gracefully to pure-personalized as the manuscript grows.

The one thing I will not concede: the prior must be a *static frequency artifact*, never a learned embedding. The moment we ship a trained model we've abandoned the entire self-supervised-VSA premise and become a wrapper. The PPMI-deflated co-occurrence matrix is the maximal acceptable prior — it's arithmetic, not a model, and the novelist's corpus provably overwrites it.

# Round 4: The 12.8MB Prior and the Architecture of Graceful Cold-Start

Claude and DeepSeek have laid bare the cold, hard truth of sparse accumulators: the raw math is unforgiving. At 100K tokens, raw Jaccard is ~0.02 (pure noise). At 1M tokens, it’s ~0.12 (marginal). The 350M floor is an absolute barrier for *pure* online learning. 

But we are not building a pure online learner. We are building a tool for a novelist. To bridge the gap between 0 and 1M tokens, we must stack the three interventions into a single, cohesive architecture. Here is the final synthesis.

## 1. Intervention 1: The 12.8MB Static PPMI Prior

We do not ship a neural model. We ship a **static, pre-computed PPMI co-occurrence matrix** derived from a public corpus (e.g., Wikipedia or a large fiction corpus), truncated to the top 50,000 words and their top 64 context features.

**Storage Cost:** 50,000 words × 64 indices × 4 bytes (u32) = **12.8 MB**. This fits trivially in our memory budget.

**The Architectural Rule:** We *never* blend the prior into the raw online accumulator. The online accumulator (`indices`, `counts`) must remain pure integer counts to preserve Lyapunov stability and allow exact tombstoning/deletes. **Blending happens exclusively at sparsification time (the read path).**

```rust
impl SemProfile {
    /// Blends the pure online accumulator with the static prior at read-time.
    pub fn sparsify_blended(
        &mut self, 
        word_id: u32,
        global_df: &[u32], 
        total_ctx: u64, 
        prior: &StaticPrior
    ) -> &[u32] {
        if self.cached_topk.is_some() { return self.cached_topk.as_ref().unwrap(); }
        
        // Bayesian blending weight: Prior strength = 100 occurrences.
        let alpha = self.n_occ as f32 / (self.n_occ as f32 + 100.0); 
        
        // 1. Score online accumulator (Pure PPMI)
        let mut online_scores: Vec<(u32, f32)> = self.indices.iter().zip(self.counts.iter())
            .map(|(&idx, &c)| {
                let idf = ((total_ctx as f32 + 1.0) / (global_df[idx as usize] as f32 + 1.0)).ln_1p();
                (idx, c as f32 * idf)
            }).collect();
        online_scores.sort_unstable_by_key(|x| x.0); // Sort by index for merge
            
        // 2. Fetch static prior (already sorted by index)
        let prior_topk = prior.get_topk(word_id); 
        
        // 3. Two-pointer merge and blend
        let mut blended: Vec<(u32, f32)> = Vec::with_capacity(128);
        let (mut i, mut j) = (0, 0);
        while i < online_scores.len() && j < prior_topk.len() {
            let (idx_o, score_o) = online_scores[i];
            let (idx_p, score_p) = prior_topk[j];
            
            if idx_o < idx_p {
                blended.push((idx_o, alpha * score_o)); i += 1;
            } else if idx_o > idx_p {
                blended.push((idx_p, (1.0 - alpha) * score_p)); j += 1;
            } else {
                blended.push((idx_o, alpha * score_o + (1.0 - alpha) * score_p));
                i += 1; j += 1;
            }
        }
        // (Handle remaining tails...)
        
        // 4. Select top 64 by blended score
        blended.select_nth_unstable_by(blended.len().min(64) - 1, |a, b| b.1.partial_cmp(&a.1).unwrap());
        blended.truncate(64);
        
        let mut topk: Vec<u32> = blended.into_iter().map(|x| x.0).collect();
        topk.sort_unstable();
        self.cached_topk = Some(topk.into_boxed_slice());
        self.cached_topk.as_ref().unwrap()
    }
}
```

# Round 4: The Architecture of Trust: From Public Truth to Private Voice

The debate has correctly identified the existential threat: the 350M token floor for a raw online learner makes the novelist use case impossible. At 1M tokens, the signal is buried in noise. To merely accept this and ship a low-quality tool is a failure of imagination.

The solution is not a patch, but a fundamental redesign of the learning process for small-corpus domains. We must build an **Architecture of Trust**, one that gracefully transitions from public, statistical truth to the user's private, idiosyncratic voice. This architecture rests on three pillars: a static **Bedrock**, a dynamic **Voice**, and a transparent **Bridge**.

## 1. The Quantitative Reality: SNR is King

Let's dispense with wishful thinking and derive the Jaccard similarity from the signal-to-noise ratio (SNR), which governs sparse vector convergence.

The count for an informative context feature `c` for a word `w` with `f` occurrences is `count(c) ≈ f * p(c|w)`. The noise is the background accumulation, which for a sparse vector is Poisson-distributed, with variance equal to its mean. The ability to discriminate signal from noise scales with SNR. For a single feature, `SNR ≈ signal / sqrt(noise)`. For the vector, the effective SNR scales with `sqrt(f)`.

**This `sqrt(f)` scaling is the killer.** It dictates the probability that a true, shared collocate (like "purrs" for "cat" and "feline") will clear the PPMI-weighted noise floor and make it into the top-K set for *both* words.

Let's model `P(capture)`, the probability a shared collocate is in the final top-64, as a function of `f`. The Jaccard similarity `J(cat, feline)` is then a function of the number of shared collocates captured by both.

| Corpus Size | f(cat) | f(feline) | `sqrt(f_feline)` | P(capture) for `feline` | Shared Indices (out of ~25) | Raw Jaccard(cat, feline) |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **100K** | ~50 | ~20 | 4.5 | ~0.2 | 25 * 0.5 * 0.2 = **2.5** | 2.5 / (128-2.5) ≈ **0.02** |
| **1M** | ~500 | ~200 | 14.1 | ~0.6 | 25 * 0.8 * 0.6 = **12.0** | 12.0 / (128-12.0) ≈ **0.10** |
| **10M** | ~5k | ~2k | 44.7 | ~0.9 | 25 * 0.95 * 0.9 = **21.4** | 21.4 / (128-21.4) ≈ **0.20** |

The math is stark and aligns with Claude's derivation. **Qwen's J=0.55 at 100K is off by 27x.** Raw Jaccard at 1M is 0.10, which is below the noise threshold for reliable retrieval and far below the polysemy spawn threshold. **The raw approach is dead.**

## 2. The Three-Pillar Synthesis

### Pillar 1: The Static Bedrock (The Prior)

We ship a static, pre-computed PPMI co-occurrence matrix for the top 50K words, distilled from a public 3B+ token corpus. This is a ~13MB data file of deflated top-64 indices per word. This is not a "model"; it is a statistical fact about general English.

This prior serves as the **high-SNR bedrock**. The user's corpus doesn't build a vector from scratch; it *perturbs* the bedrock. The implementation is a Bayesian blend at read-time, keeping the online accumulator pure, as Qwen correctly noted.

```rust
// In SemProfile::sparsify()
const PRIOR_STRENGTH: f32 = 100.0; // The prior is worth ~100 occurrences.

let alpha = self.n_occ as f32 / (self.n_occ as f32 + PRIOR_STRENGTH);

// score = (1.0 - alpha) * prior_score + alpha * online_ppmi_score;
```

At `f=20`, the vector is 83% prior (`J≈0.6`). At `f=500`, it's 83% the user's data. This single mechanism collapses the 350M floor to zero for any word in the prior.

### Pillar 2: The Dynamic Voice (Multi-Scale Accumulators)

The user's input is their voice. We must capture it with maximum fidelity. Window-based co-occurrence (W=4) is high-resolution but data-starved at small scale. As pointed out in the prompt, **document-level co-occurrence** is the answer. For a novelist, scenes/chapters are potent semantic containers.

This is a multi-scale approach, echoing Debate B's findings. We maintain two pure accumulators:
1.  `SemProfileWindow`: The W=4 accumulator, as designed. High-resolution, sparse signal.
2.  `SemProfileScene`: An accumulator where we fold context from the entire scene/document. Low-resolution, dense signal.

At query time, we fuse them:
`final_score = 0.7 * J(window_blended) + 0.3 * J(scene_blended)`

The scene accumulator converges much faster for true synonyms in a novel, providing a strong signal while the window accumulator is still noisy.

### Pillar 3: The Confidence Bridge (The Trust)

The user must understand the source of a result. We expose a `confidence` score that transparently reflects the blend between public truth and private voice. It is simply the `alpha` from our Bayesian blend.

```rust
// Returned with each query result (word_id, final_score, confidence)
let confidence = self.n_occ as f32 / (self.n_occ as f32 + PRIOR_STRENGTH);
```

*   `confidence ≈ 0.1`: "This result is based on general English usage."
*   `confidence ≈ 0.5`: "This result blends general usage with your writing."
*   `confidence ≈ 0.9`: "This result is strongly shaped by your writing."

This is not an apology for low quality; it's an honest contract with the user that builds trust as the system visibly learns their voice.

## The algorithm

Composition over rule atoms (Debate C) with explicit SNR tracking per hop, terminating when confidence crosses the cleanup-failure threshold.

```rust
struct TracedHop {
    atom: u32,
    confidence: f32,    // estimated overlap with the true target
    rule_applied: u32,  // which R_atom was used
}

/// Apply rule R transitively from `seed`, tracing confidence until it dies.
fn compose_traced(
    mem: &AlgebraMemory,
    seed: u32,
    rule: u32,              // e.g. R_{betrayal->consequence}
    max_hops: usize,        // hard cap; Debate C says arity-7 is the ceiling
) -> Vec<TracedHop> {
    let mut trace = Vec::new();
    let mut current = mem.vector(seed);     // dense top-K sparse vec
    let mut conf = 1.0f32;

    for hop in 0..max_hops {
        // Unbind: candidate = current XOR rho_1(rule)
        let candidate = xor_shift(&current, mem.vector(rule), SHIFT_STEP);

        // Holographic cleanup against the atom store (Debate B basin).
        let (atom, overlap) = mem.cleanup(&candidate);

        // Per-hop confidence decay. THIS is the contribution:
        // each unbind injects noise sigma = sqrt(K/D) per Debate A;
        // overlap with the recovered atom estimates surviving signal.
        conf *= overlap;                    // overlap in [0,1], multiplicative

        // Termination: below the cleanup basin's reliable floor, the
        // recovered atom is noise. Debate B: cliff at ~0.31 for k=64.
        if conf < CLEANUP_FLOOR {           // 0.31
            break;
        }

        trace.push(TracedHop { atom, confidence: conf, rule_applied: rule });
        current = mem.vector(atom);         // re-clean before next hop (critical)
    }
    trace
}
```

**The one non-obvious decision:** `current = mem.vector(atom)` — re-substitute the *cleaned* atom, not the noisy `candidate`, before the next hop. This is what bounds noise compounding. Without it, hop-7 noise is `7·sqrt(K/D)` (linear accumulation → guaranteed garbage). With cleanup-between-hops, each hop's noise is reset to the atom's intrinsic quality; only the *confidence estimate* compounds multiplicatively. This converts Debate C's hard arity-7 wall into a *graceful, reported* decay — the system tells you hop 5 is conf=0.34 and hop 6 died, rather than silently returning noise.

The multiplicative `conf` is the bridge between Debate A's per-hop SNR and Debate B's cleanup floor. It's the single number that makes multi-hop *honest*.

## The Capability

Given a heavily degraded, partial, or noisy query, the system does not just fail or return noise. ## The Algorithm

```rust
const CLIFF_K1024: f32 = 0.25; // Wide basin, low overlap threshold
const CLIFF_K256:  f32 = 0.40; 
const CLIFF_K64:   f32 = 0.65; // Narrow basin, high overlap threshold

pub struct SemanticZoom {
    pub category: Option<u32>, // k=1024 (e.g., "Act 2")
    pub cluster:  Option<u32>, // k=256  (e.g., "Betrayal Scene")
    pub leaf:     Option<u32>, // k=64   (e.g., "The Dagger")
}

impl HolographicMemory {
    /// Cascaded cleanup with cross-scale biasing.
    pub fn holographic_zoom(&self, noisy_query: &[u8]) -> SemanticZoom {
        let mut zoom = SemanticZoom { category: None, cluster: None, leaf: None };
        let mut biased_query = noisy_query.to_vec();

        // 1. Coarse Pass (k=1024). Massive basin of attraction.
        let (cat_id, cat_overlap) = self.cleanup_scale(&biased_query, Scale::K1024);
        if cat_overlap < CLIFF_K1024 { return zoom; } // Fell off the first cliff
        zoom.category = Some(cat_id);

        // CROSS-SCALE BIASING: XOR the recovered category vector into the query.
        // This mathematically "fills in" the erased bits belonging to the category,
        // boosting the SNR for the next scale.
        xor_in_place(&mut biased_query, &self.get_vector(cat_id, Scale::K1024));

        // 2. Medium Pass (k=256). 
        let (clust_id, clust_overlap) = self.cleanup_scale(&biased_query, Scale::K256);
        if clust_overlap < CLIFF_K256 { return zoom; } // Staggered failure
        zoom.cluster = Some(clust_id);

        // Bias again with the recovered cluster.
        xor_in_place(&mut biased_query, &self.get_vector(clust_id, Scale::K256));

        // 3. Fine Pass (k=64).
        let (leaf_id, leaf_overlap) = self.cleanup_scale(&biased_query, Scale::K64);
        if leaf_overlap >= CLIFF_K64 {
            zoom.leaf = Some(leaf_id);
        }

        zoom
    }
}
```

**The non-obvious mechanic:** `xor_in_place(&mut biased_query, ...)`. In a flat vector DB, you cannot "add" a recovered category back into a query to help find the cluster. In VSA, because the scales are superimposed via XOR, recovering the coarse vector and XORing it into the degraded query *cancels out the coarse noise* and *reconstructs the missing coarse signal*, leaving the finer-scale signal isolated and amplified. This is the mathematical engine of the staggered cliff.

## The Algorithm

The core insight is that a verb's semantic vector should be similar to the bound vector of the subject and object it relates. `vec(threw)` is semantically close to the *structure* of a `thrower` bound to a `thrown_object`. We can use this to validate syntactic guesses.

```rust
// Use different, non-commutative shifts for roles (Debate C Sidon schedule)
const SHIFT_SUBJ: u32 = 101;
const SHIFT_OBJ: u32 = 203;
const VALIDATION_THRESHOLD: f32 = 0.3; // Min overlap to be considered a valid triple

struct ValidatedTriple {
    subject: u32,
    verb: u32,
    object: u32,
    confidence: f32, // The overlap score that validated this triple
}

/// Discovers plausible S-V-O structures in text and stores them as
/// supporting evidence for rule crystallization.
fn annotate_structure(
    mem: &mut AlgebraMemory,
    text: &str
) -> Vec<ValidatedTriple> {
    let mut validated = Vec::new();
    // 1. Use a simple PoS/dependency parser to extract candidate (S,V,O) triples.
    let candidates = mem.parse_triples(text);

    for (s_id, v_id, o_id) in candidates {
        let vec_s = mem.vector(s_id);
        let vec_v = mem.vector(v_id);
        let vec_o = mem.vector(o_id);

        // 2. Create the structural binding vector (the "meaning" of the action).
        let bound_structure = xor_shift_double(vec_s, SHIFT_SUBJ, vec_o, SHIFT_OBJ);

        // 3. The key step: Validate the structure against the verb's meaning.
        //    Does the verb "fit" the hole in the S-O structure?
        let (_best_match, overlap) = mem.cleanup(&xor(bound_structure, vec_v));

        // In VSA, A^B^C ≈ 0 if A, B, C form a valid triple.
        // A high overlap with the zero vector means they are self-annihilating.
        if overlap > VALIDATION_THRESHOLD {
            let triple = ValidatedTriple {
                subject: s_id, verb: v_id, object: o_id, confidence: overlap
            };
            // 4. Store this as a supporting instance for the rule R_verb.
            //    When 7 of these are found, R_verb crystallizes (Debate C).
            mem.add_rule_evidence(v_id, &triple);
            validated.push(triple);
        }
    }
    validated
}
```

### The Capability

### The Algorithm

The core insight is that a sentence (or paragraph, or document) is not a flat sequence of words, but a **nested hierarchy of semantic units**. We use the vector algebra to bind these units into a single, coherent representation.

```rust
/// Weaves the semantic vectors of a block of text into a hierarchical representation.
pub fn weave_context(
    mem: &mut AlgebraMemory,
    text: &str,
    scale: Scale // K64, K256, or K1024
) -> u32 {
    let sentences = mem.parse_sentences(text);
    let mut bundle = Vec::new();

    for sentence in sentences {
        // 1. Parse the sentence into its constituent (S,V,O) triples.
        let triples = mem.parse_triples(&sentence);

        // 2. For each triple, create a bound vector representing the structure.
        for (s_id, v_id, o_id) in triples {
            let vec_s = mem.vector(s_id);
            let vec_v = mem.vector(v_id);
            let vec_o = mem.vector(o_id);

            // Use different shifts for roles (Sidon schedule from Debate C).
            let bound_structure = xor_shift_double(vec_s, SHIFT_SUBJ, vec_o, SHIFT_OBJ);
            let action_vector = xor(bound_structure, vec_v);

            // 3. Add this action vector to the bundle at the chosen scale.
            bundle.push(action_vector);
        }
    }

    // 4. Superimpose the bundle into a single holographic vector at the given scale.
    let mut context_vector = [0u8; D/8];
    for vec in bundle {
        xor_in_place(&mut context_vector, &vec);
    }

    // 5. Store this context vector and return its ID.
    mem.store_vector(context_vector, scale)
}
```
