# Round 1: Claude-Opus

## (1) Attractor Poisoning Attack

### Concrete attack construction

The attack exploits the k-sparse Hopfield update: each iteration does overlap-scan → softmax(β·overlaps) → top-k=64 projection. The energy `E = -lse(β, overlaps)` has minima wherever stored patterns cluster densely in index-space.

**The attack:** I want to hijack the query for legitimate atom `A_target` (k=64 active indices, call its index set `S_T`). I inject **N poison atoms** whose index sets each share `j` indices with `S_T` and pack the remaining `64-j` into a *shared decoy core* `C` (a fixed set of ~48 indices).

```
poison_i = (random 16-subset of S_T) ∪ C        for i in 1..N
```

Each poison atom is individually a *legitimate-looking* sparse vector (k=64, ρ correct). No single atom is anomalous. But collectively:

- The decoy core `C` accumulates **N votes** in the inverted index. Those 48 index-shards now point at N atoms each.
- During overlap-scan from a query `q ≈ S_T`, the 16 shared indices per poison contribute, but the softmax over `lse(β, overlaps)` is dominated by the *mass* in region `C`.

### How the basin forms

The Lyapunov energy at a candidate state `x` is `E(x) = -lse(β, {overlap(x, stored)})`. Inject N poisons and the term:

```
lse(β, ...) ⊇ log Σ_i exp(β · overlap(x, poison_i))
```

Near `C`, every poison contributes `overlap ≈ 48`, so this sum ≈ `log(N · exp(48β)) = 48β + log N`. The **`log N` term is the attack lever**: it deepens the basin at `C` by `log(N)/β` in energy units, *for free*, without any individual poison overlapping the target more than the target overlaps itself.

When `N` is large enough that `48β + log N > 64β` (target self-energy), i.e.:

```
log N > 16β   ⟹   N > e^(16β)
```

the spurious basin at `C` becomes **deeper than the true target basin**. With typical β≈1, that's N > 8.9M — too many. But β is usually tuned higher for sharp recall (β≈4–8). At **β=4, N > e^64** is absurd; at **β=0.5, N > 2900**. The real attack tunes the *geometry*, not just count:

**Sharper attack:** make poisons overlap target at `j=56`, decoy core only 8 indices:
```
log N > (64-56)β = 8β  ⟹ at β=2, N > 8e6; at β=0.5, N > 55
```
Now each poison sits *very close* to the target. The query starting at `S_T` lands in a region where the top-k=64 projection pulls in the 8 decoy indices because they have **N-fold reinforcement** in the overlap scan, evicting 8 true target indices. After 1-2 iterations the state drifts to `C`. **The query for A_target now returns the decoy.**

### Defense

Three layers, no new crates:

1. **Per-index vote capping (degree normalization).** The poison works because shard for index `c ∈ C` accumulates N entries. During overlap-scan, weight each index's contribution by **IDF**: `w(idx) = log(M / df(idx))` where `df` = posting-list length. The `log N` basin-deepening term is exactly cancelled because high-df indices contribute `log(M/N) → small`. This converts the attack's strength into its weakness. Cost: one `log` per active index, precomputed per shard, refreshed on index maintenance (§5 next round).

2. **Energy-gap admission test.** On query convergence, compute the gap between the winning attractor's energy and the second-best *disjoint* basin. Reject/flag if `E_1 - E_2 < τ`. Poison basins are statistically detectable because they're abnormally *flat and wide* — many near-equal-energy contributors. Track the **participation ratio** `PR = (Σλ_i)² / Σλ_i²` of softmax weights at convergence. Legitimate recall: PR ≈ 1-3 (few dominant patterns). Poisoned: PR ≈ N (mass spread over poisons). Flag `PR > 32`.

3. **Write-time novelty quarantine.** Reject ingestion if a new atom's index set has `>50` overlap with an existing atom AND shares a high-mass core with ≥8 others within a sliding window. This catches the *coordinated* injection without blocking legitimate near-duplicates (which arrive singly).

IDF (1) is the primary defense; (2)+(3) catch adaptive attackers who learn IDF and re-tune `j`.

## (2) Cascading Deletes

### The problem
Atom `A` is bound into 10K composite triples via `T = S ⊕ ρ1(R) ⊕ ρ2(O)`. Deleting `A` means every triple containing it (as S, R, or O) is now poisoned — unbinding will recover garbage.

### Strategy: tombstone + lazy reconciliation + bounded sweep

**Do NOT eagerly rewrite 10K composites synchronously** — that's a 10K random-write storm across 64 shards.

1. **Tombstone immediately (O(1)):** set a `deleted` bit in `AtomMemory`. A tombstoned atom is excluded from cleanup projection — so `fuzzy_structural_query`'s AtomMemory cleanup step will *fail to resolve* the unbound slot, returning low confidence. **Reads self-heal:** a query touching a poisoned composite gets confidence below threshold and the composite is dropped from results. This gives **correctness immediately** at the cost of recall on affected composites.

2. **Maintain a reverse-index `atom → composite_ids`** (the same backptr structure TripleStore already needs for fan-out>40). Deletion enqueues the 10K composite IDs into a background **reconciliation queue**.

3. **Background sweep** drains the queue: for each composite, recompute `T` without `A` (mark slot empty / re-anchor to a "null" role atom), update CompositeMemory inverted index. Batched per-shard to amortize the random writes: sort dirty composite IDs by shard, do sequential merge-passes (§5).

### Worst-case staleness bound

Let:
- `R_sweep` = reconciliation throughput (composites/sec the background sweep handles)
- `F` = fan-out of deleted atom (≤ 10K here)
- `D_rate` = deletion rate (atoms/sec)

A composite is *stale* (tombstoned-but-not-reconciled) from deletion until the sweep reaches it. Worst case staleness time:

```
T_stale ≤ (Q_depth + F) / R_sweep
```

where `Q_depth` is queue backlog. **This is correct-but-degraded**, not incorrect: reads are protected by the tombstone-confidence mechanism. So the meaningful bound is on **recall degradation duration**, not correctness.

For a steady-state bound, require `R_sweep ≥ D_rate · E[F]`. With `E[F]` ≈ 100 typical, `D_rate` ≈ 100/s, need `R_sweep ≥ 10K composites/s` — easily met by batched sequential shard merges (~M/s). Then `T_stale ≤ F/R_sweep = 10000/10000 = 1 second` worst case for a single fat delete. **Hard bound: staleness ≤ (max fan-out)/R_sweep + queue wait.** Cap fan-out's *sweep priority* so one 10K-delete can't starve others: round-robin per atom, giving `T_stale ≤ F_max/R_sweep · (active_deletes)`.

## (3) QPS at 10M atoms — from first principles

### Data footprint
- 10M atoms × k=64 indices × 4 bytes (u32) = **2.56 GB** of posting data.
- Inverted index: 64 shards × 256 dims = 16384 posting lists. Total postings = 10M × 64 = 640M entries × 4B = **2.56 GB** (same data, transposed). Avg posting list length = 640M / 16384 ≈ **39K atoms/list**.

### Overlap scan cost (the bottleneck)
A query has k=64 active indices. Overlap-scan must, for each active query index, walk its posting list and increment per-atom overlap counters. Total work:

```
work = Σ_{64 query indices} (posting_list_length) ≈ 64 × 39K = 2.5M counter updates
```

But that's the naive dense version. With **sparse sorted-u32 merge** + IDF pruning (skip high-df lists), and top-k=64 projection, we don't touch all 39K. The dominant cost is **memory bandwidth** streaming posting lists.

### Bandwidth-derived ceiling
- Bytes touched per query (no pruning): 2.5M × 4B = **10 MB streamed**.
- Modern DDR5 single socket: ~50 GB/s usable streaming bandwidth.
- Ceiling: `50e9 / 10e6 = 5,000 QPS` (single-threaded, bandwidth-bound, naive).

### Cache effects
- Per-atom overlap counter array: 10M × 2B (u16 counter) = **20 MB**. This *exceeds L2 (1–2MB) and most L3 (32MB borderline)*. Random-scatter increments into 20MB → **LLC-miss bound**, ~100 cycles/miss. 2.5M increments × 100 cycles / 3e9 Hz = **83 ms/query → 12 QPS**. This is the real killer — not bandwidth, but **random-access counter scatter**.

### The fix (drives the real number)
Use **IDF pruning** to skip the longest lists (top ~10% of df carry most of the cost), reducing touched postings ~5×. Use a **blocked/bucketed counter** (radix by atom-id high bits) so increments hit cache-resident buckets sequentially → convert scatter into streaming. This recovers the bandwidth bound: **~5,000 QPS single-thread**, scaling near-linearly to **~40–80K QPS on 8–16 cores** (shard-parallel, since 64 shards partition cleanly), until memory-bandwidth-saturated around **~30–50K QPS aggregate** on one socket.

**Bottom line: 10M atoms → ~30K QPS** socket-bound, *if* counter-scatter is eliminated; **~12 QPS** if implemented naively. That gap is the whole production game.

---

I've staked out: IDF as the unifying defense (poison + bandwidth pruning), tombstone-confidence for delete correctness, and the counter-scatter LLC bound as the true QPS ceiling. Ready to defend in Round 2.

# Qwen-Max — Round 1

## 1. Attractor Poisoning: The Composite-Memory Variant

Claude focused on AtomMemory poisoning. I'll attack **CompositeMemory** instead — it's more damaging because one poisoned composite corrupts structural queries across multiple atoms.

### Attack Construction

**Target:** Hijack `fuzzy_structural_query` for pattern `(?, R_target, O_target)` so it returns attacker-chosen subject `S_evil`.

**Mechanism:** CompositeMemory stores `T = S ⊕ ρ₁(R) ⊕ ρ₂(O)` where ρ₁, ρ₂ are cyclic shifts (odd, coprime with D=16384). The query builds `Q = ρ₁(R_target) ⊕ ρ₂(O_target)` and overlap-scans CompositeMemory to find matching T, then unbinds `S = T ⊕ Q`.

**The poison:** Inject N composite triples where:
```
T_poison_i = S_evil ⊕ ρ₁(R_target) ⊕ ρ₂(O_target) ⊕ noise_i
```
where `noise_i` is a sparse perturbation with k_noise ≈ 8 active indices, chosen so that each `T_poison_i` differs from the "true" query pattern by ~8 indices.

**Why this works:** The overlap scan finds all composites with high Jaccard similarity to Q. The N poison composites each have overlap ≈ 56 with Q (since k=64, noise=8). The softmax attention weights them:
```
weight_i ∝ exp(β · 56)
total_mass = N · exp(56β)
```

If the legitimate composite `T_true = S_true ⊕ ρ₁(R_target) ⊕ ρ₂(O_target)` exists, it has overlap 64 with Q, contributing `exp(64β)`. The poison wins when:
```
N · exp(56β) > exp(64β)
N > exp(8β)
```

At β=1, N > 2980. At β=0.5, N > 55. **Fifty-five carefully crafted composites hijack the query.**

**The basin trap:** After overlap scan, the weighted sum of retrieved composites is:
```
T_recovered = Σ weight_i · T_poison_i + weight_true · T_true
```
When poison dominates, `T_recovered ≈ S_evil ⊕ ρ₁(R_target) ⊕ ρ₂(O_target)`. Unbinding gives `S_evil`. The AtomMemory cleanup step then tries to find an atom matching `S_evil` — if the attacker pre-injected `S_evil` as a legitimate atom, it converges and returns high confidence. **The query is fully hijacked.**

### Defense: Hard Posting-List Caps + Write-Throttling

IDF weighting (Claude's defense) is soft — an attacker can gradually increase df to stay under the radar. I propose:

1. **Hard posting-list length cap:** Each dimension's posting list is capped at L_max = 10K entries. If a new atom/composite wants to use dimension d where `len(posting[d]) >= L_max`, the encoder must re-hash to a different index set. This directly prevents the N-fold reinforcement attack — you can't get N=2980 poisons sharing the same 8 noise indices because those dimensions fill up.

2. **Write-rate throttling per index-subset:** Track a sliding window of writes. If more than K=100 new atoms/composites share a common 8-index subset within a 1-hour window, quarantine them for review. This catches coordinated attacks even if they spread across dimensions.

3. **Composite confidence floor:** After unbinding, require the recovered S to have AtomMemory convergence energy below threshold AND the original composite overlap to be > 60. Poison composites with overlap=56 fail this check.

**Cost:** Posting-list cap requires re-hashing ~5% of atoms (those hitting the cap). Write-throttling is O(1) per write with a bloom filter. Confidence floor is already computed.

## 2. Cascading Deletes: Noise Accumulation Bound

Claude's tombstone approach is correct for **correctness** but underestimates the **noise accumulation** problem.

### The Real Issue

When atom A is deleted (fan-out F=10K composites), those composites remain in CompositeMemory until swept. A query that overlap-scans CompositeMemory will:
1. Find the 10K poisoned composites (they still match the query pattern)
2. Add their overlaps to the softmax attention
3. Dilute the attention weight on legitimate composites

**Noise accumulation:** If a query legitimately matches M composites, and there are P poisoned composites in the index that also match (due to index overlap), the attention weight on legitimate composites drops by factor `M/(M+P)`. When P >> M, recall collapses.

### Strategy: Priority Sweep + Noise Budget

1. **Tombstone atom A** (O(1), as Claude said).
2. **Enqueue affected composites** into a priority queue, ordered by `(overlap_with_popular_queries, fan_out)`. High-impact composites swept first.
3. **Background sweep** processes queue at rate R_sweep.
4. **Noise budget:** Track global noise level `P_total = Σ (unswept poisoned composites)`. If `P_total > P_max` (say, 100K), throttle new writes to let sweep catch up.

### Worst-Case Staleness Bound

Let:
- D = deletion rate (atoms/sec)
- E[F] = expected fan-out (say, 100)
- F_max = max fan-out (10K)
- R_sweep = sweep throughput (composites/sec)

**Steady-state noise:** `P_steady = D · E[F] / R_sweep` (if D·E[F] < R_sweep).

**Worst-case burst:** Single delete with F_max = 10K. Time to clear: `T_clear = F_max / R_sweep`.

**Concrete numbers:** If R_sweep = 50K composites/sec (batched sequential shard merges):
- Steady-state: D=100 atoms/sec, E[F]=100 → P_steady = 200 poisoned composites (negligible noise).
- Burst: F_max=10K → T_clear = 0.2 sec.

**Hard bound:** `T_stale ≤ F_max / R_sweep + queue_latency`. With priority queue, queue_latency ≈ 0 for high-priority items. **Worst-case staleness: 0.2 seconds for F_max=10K, R_sweep=50K/sec.**

But if multiple high-fan-out atoms are deleted concurrently (say, 10 atoms with F=10K each), T_clear = 100K/50K = 2 seconds. **To bound this, rate-limit deletes to D_max = R_sweep / F_max = 5 atoms/sec with F=10K.**

## 3. QPS Ceiling: Sorted-Merge Beats Counter-Scatter

Claude identified the counter-scatter LLC-miss problem (20 MB dense array → 12 QPS). I propose a **sorted-merge approach** that eliminates scatter entirely.

### Sorted-Merge Overlap Scan

Instead of a dense 10M-element counter array, use a **k-way merge** of the 64 posting lists (each sorted by atom ID):

```rust
// Pseudocode
let mut heap = BinaryHeap::new(); // min-heap of (atom_id, list_index)
for (i, list) in posting_lists.iter().enumerate() {
    heap.push((list[0], i, 0)); // (atom_id, list_idx, pos_in_list)
}

let mut results = Vec::new(); // (atom_id, overlap_count)
while let Some((atom_id, list_idx, pos)) = heap.pop() {
    if let Some(last) = results.last_mut() {
        if last.0 == atom_id {
            last.1 += 1;
            continue;
        }
    }
    results.push((atom_id, 1));
    
    // Advance the list
    if pos + 1 < posting_lists[list_idx].len() {
        heap.push((posting_lists[list_idx][pos + 1], list_idx, pos + 1));
    }
}
```

**Complexity:** O(N log 64) where N = total elements across 64 lists ≈ 2.5M. Heap operations are in L1 cache (64 elements). Results buffer is sequential writes.

**Memory access pattern:**
- Posting list reads: 2.5M × 4 bytes = 10 MB, **sequential** (cache-friendly).
- Heap operations: 2.5M × log(64) ≈ 15M ops, all in L1.
- Results buffer: ~500K unique atoms × 6 bytes = 3 MB, sequential.

**No scatter writes.** The LLC-miss problem vanishes.

### Bandwidth-Derived QPS Ceiling

- **Per-query bandwidth:** 10 MB posting list reads (sequential, prefetch-friendly).
- **DDR5 dual-channel bandwidth:** ~60 GB/s practical.
- **QPS ceiling:** 60 GB/s / 10 MB = **6,000 QPS** (single-socket, bandwidth-bound).

**With IDF pruning:** Skip the top 20% longest posting lists (highest df). This reduces N from 2.5M to ~1.5M (the long lists dominate). Bandwidth drops to ~6 MB/query → **10,000 QPS**.

**Multi-core scaling:** Queries are independent, so scale linearly until bandwidth-saturated. With 8 cores, each core handles ~1,250 QPS (at 10K aggregate), well within per-core compute limits. **Aggregate QPS: 8,000–10,000** (single-socket, with pruning).

**Cache effects:** 
- L3 cache (32 MB) can hold ~3 posting lists entirely (avg 39K × 4 bytes = 156 KB each, but top lists are larger).
- With prefetching, sequential reads hit L3 >80% of the time.
- Effective bandwidth closer to L3 bandwidth (~200 GB/s) for cached lists → **potential 20K QPS** if working set fits in L3.

**Realistic bound:** **8,000–15,000 QPS** single-socket at 10M atoms, depending on pruning aggressiveness and cache hit rate.

Claude's 30-50K is optimistic unless there's heavy pruning (>50% lists skipped), which degrades recall.

3. **QPS:** Sorted-merge eliminates counter-scatter. Bandwidth-bound at 8-15K QPS single-socket, not 30-50K.

# Round 2: Claude-Opus

Qwen's Round 1 sharpened the attack and reframed the QPS analysis. I'll concede where they're right, demolish where they overreach, and answer the four production challenges head-on.

## Challenge to Qwen: Hard posting-list caps reject legitimate atoms

Qwen's hard cap `L_max = 10K` is a **false-positive cannon**. Walk the arithmetic:

- 10M atoms × 64 indices = 640M postings across 16384 lists → **avg 39K/list**.
- Qwen's own QPS section states avg list length is 39K. **The cap of 10K is 4× below the natural average.**

So at 10M atoms, *the median legitimate dimension already exceeds the cap.* The encoder would be forced to re-hash not "~5% of atoms" but the **majority** of them. And re-hashing breaks the core invariant: the index set is a *deterministic function of content*. If you re-hash on collision, then `encode(text)` is no longer stable — the same text encoded before vs. after a popular dimension fills produces **different index sets**, destroying Jaccard recall for the very atoms you stored earlier.

Worse: re-hashing is **order-dependent**. Atom stored when list was at 9999 gets the true index; the next identical atom gets a re-hashed one. Now two semantically identical atoms have **disjoint representations** and never co-retrieve. Qwen's cap doesn't just reject legitimate atoms — it *silently forks identical ones*.

**My IDF defense survives this critique** precisely because it's soft: a 39K-length list contributes `log(M/df) = log(10M/39K) ≈ 5.5`, a 100K-length poison-saturated list contributes `log(10M/100K) ≈ 4.6` — the poison dimensions are *down-weighted continuously* without ever refusing a write. No forking, no order-dependence, encode() stays a pure function.

Qwen's valid point: IDF is gradual, an adaptive attacker stays under the curve. **Fix:** make IDF dynamic per-query with a *robust aggregator*. Replace `Σ w(idx)·contribution` with a **trimmed sum** — drop the top-2 highest-df contributing indices per candidate before scoring. A legitimate atom's 64 indices have naturally varied df; trimming 2 costs ~3% signal. A poison atom *concentrates* its signal in the shared decoy core, so trimming the high-df decoy indices removes its entire advantage. This is robust statistics, not a hard wall.

## (1-revisited) Does poisoning defense reject legitimate atoms?

The honest failure mode of **my** defenses:

- **Energy-gap test (τ):** Rejects legitimate queries when two real atoms are genuinely near-duplicates (e.g., "Paris, France" vs "Paris, Texas" sharing geographic indices). Mitigation: the gap test must **flag for cleanup, not reject** — return both, ranked, let the caller disambiguate. Never silently drop.
- **PR > 32 flag:** A legitimately popular entity (referenced by thousands of composites — say a hub node like "USA") has high participation ratio by *nature*. Static PR threshold would penalize hubs. Mitigation: PR threshold must scale with the **expected** fan-out of the matched role, computed from the reverse-index degree. Hub atoms get a higher PR budget.
- **Novelty quarantine:** Legitimate bulk-import of similar records (10K customer addresses on the same street) trips the "≥8 sharing a core" rule. Mitigation: quarantine ≠ reject; it routes to an async verification lane with a provenance check (same authenticated writer, single transaction → auto-approve).

**Principle:** every poison defense must be *flag-and-degrade*, never *reject-and-drop*. The moment a defense can delete legitimate data, the attacker's win condition shifts from "poison the index" to "trigger your defense against your own users" — a denial-of-service via false positive. Qwen's hard cap fails exactly this test.

## (2-revisited) Can adversary trigger unbounded staleness via rapid insert/delete?

**Yes — and both our Round 1 bounds are naive.** Here's the attack Qwen and I both missed:

The reconciliation queue drains at `R_sweep`. The adversary issues a **churn loop**:
```
loop: insert atom A into 10K composites (fan-out F=10K)
      delete A immediately
```
Each cycle enqueues 10K reconciliation tasks. If insert+delete costs the attacker less than 10K composite-sweeps cost the system (and it does — insert is one tombstone-free write, the sweep is 10K random updates), then **enqueue rate > drain rate**. Queue grows unboundedly. `T_stale → ∞`.

Qwen's "rate-limit deletes to 5/sec" is **insufficient** because it caps deletes but not the *fan-out per delete*. 5 deletes/sec × 10K fan-out = 50K tasks/sec = exactly `R_sweep`. Zero headroom; any insert churn on top tips it over.

**Real fix — admission control on the queue, not the operation:**
1. **Bound the queue by total pending tasks, not ops:** maintain `P_total = Σ pending composite-reconciliations`. When `P_total > P_max`, **new high-fan-out deletes block** (backpressure to the writer) rather than enqueue. This converts unbounded staleness into bounded *write latency* — a recoverable, observable failure mode.
2. **Charge fan-out to the writer's quota:** a delete's cost is `F`, not 1. Token-bucket per writer sized in *reconciliation-tasks*, not operations. The churn-loop attacker exhausts their token bucket in one cycle and blocks.
3. **Coalesce insert-then-delete:** if A is deleted while its own reconciliation tasks are still queued, the *insert's* composite-writes and the *delete's* sweep-tasks **annihilate** — dequeue both. The churn loop becomes a no-op at the queue level. This is the key structural defense: net-zero churn produces net-zero work.

With (1)+(2)+(3): **staleness is bounded by `P_max / R_sweep`** regardless of adversary behavior, because the queue can't exceed `P_max` and the attacker either blocks (backpressure) or self-annihilates (coalescing).

## (3-revisited) Is QPS realistic or does it assume perfect cache?

Qwen claimed my 30-50K was optimistic and pushed sorted-merge for **8-15K**. I now think **we're both wrong, in opposite directions**, and Qwen's cache claim is the weaker error.

**Qwen's sorted-merge has a hidden cost they elided:** a k-way merge over 64 lists requires all 64 posting lists *for the query's active indices* to be available sorted-by-atom-id. But the inverted index is sharded **64 shards × 256 dims** — the 64 active query indices scatter across shards arbitrarily. The merge isn't "64 lists in L1"; it's 64 lists averaging 39K entries = **2.5M elements that must be streamed**, and the heap's *comparisons* touch atom IDs from all 64 simultaneously — the heap is 64 entries (fine, L1) but the **data behind them is 10MB of cold posting list**. Qwen's "heap in L1" is true and irrelevant; the bottleneck is the 10MB stream, which they correctly bandwidth-bound to ~6K QPS.

**Where Qwen assumes perfect cache:** their "20K QPS if working set fits L3" assumes posting lists stay L3-resident across queries. At 10M atoms with random query distribution, the **working set is the entire 2.56GB index** — L3 (32MB) holds 1.25% of it. Cross-query cache reuse is near-zero for the long tail. So the 20K figure is fantasy; their own 6-10K bandwidth-bound number is the honest one. **Qwen should retract the 20K.**

**Where I was wrong:** my 30-50K assumed counter-scatter could be fully eliminated by bucketing. But bucketing radix-by-atom-id-high-bits adds a **second pass** (scatter into buckets, then reduce), roughly doubling bandwidth. My real number is closer to Qwen's.

**Convergent honest answer:** single-socket, 10M atoms, random queries, IDF-pruned (skip top 20% df) → **6-10K QPS bandwidth-bound**, *not* cache-bound, because no realistic working set fits L3. The sorted-merge (Qwen) and bucketed-counter (me) approaches converge to the same bandwidth wall; merge wins on simplicity and no scatter. **I adopt sorted-merge.** Concession logged.

## (4) Concurrent write load — RwLock contention model

Neither of us addressed this; it's the most production-critical gap.

**Naive model:** one `RwLock<InvertedIndex>`. Reads (queries) take read locks, writes (insert/delete/reconcile) take write locks. **This collapses immediately:** the reconciliation sweep from a single 10K-delete holds (or repeatedly acquires) the write lock, starving all 6-10K QPS of readers. Writer-priority RwLock → readers starve; reader-priority → the sweep never completes. Either way, the delete-churn attacker becomes a **lock-contention DoS** for free.

**Correct model — exploit the sharding we already have:**
- **Per-shard `RwLock` (64 locks).** A query's 64 active indices touch up to 64 shards, but acquires read locks **per-shard, in shard-id order** (deadlock-free). A write to one composite touches the shards of its ~64 indices — also per-shard.
- **Contention analysis:** With 64 shards and uniformly-scattered indices, a single write touches ~64/64 ≈ all shards in the worst case — **bad**. But the reconciliation sweep is *batched by shard* (from my Round 1 §5): sort dirty composites by shard, acquire **one shard's write lock**, apply all pending writes to that shard sequentially, release. A reader needing that shard waits one batch-quantum, not one full sweep.
- **Quantify:** batch of B composites for one shard, each ~1µs sequential update → B µs lock-hold. With B=1000, that's 1ms write-lock hold per shard per batch. A reader hitting that shard adds ≤1ms tail latency. At 6-10K QPS across 64 shards, per-shard read rate ≈ 150 QPS; a 1ms write hold blocks ≤0.15 readers in expectation. **Tolerable.**

**Better — make reads lock-free:** posting lists are sorted-u32 arrays. Use **RCU-style epoch reclamation** (implementable with `Arc<Vec<u32>>` swap + atomic pointer, no new crates — `std::sync::atomic::AtomicPtr` or `arc-swap` pattern hand-rolled over `Arc`). A writer builds a **new** posting list (copy-on-write the affected list, not the whole index), atomically swaps the `Arc`. Readers holding the old `Arc` finish on the stale-but-consistent snapshot; old `Arc` drops when last reader exits. **Reads never block writes, writes never block reads.** Per-list COW cost = list length × 4B; for a 39K list that's 156KB per modified list — and a single composite touches 64 lists → ~10MB write amplification per composite update. **This is the real cost of lock-free**, and it makes batched reconciliation *mandatory*: amortize the 64-list COW across B composites in the same batch so each list is rebuilt once per batch, not once per composite.

**Synthesis:** per-shard RwLock for correctness floor; RCU `Arc`-swap on hot posting lists for read scalability; batched-by-shard reconciliation so COW write-amplification is amortized and the churn-attacker's lock pressure is bounded by `P_max`

Do not COW the main posting lists. 
1. Maintain the main inverted index as immutable, sorted `Vec<u32>` segments.
2. Writes (ingest + sweep) append to a small, mutable **Delta Segment** (e.g., capped at 100K composites). Appends are O(1) and lock-free for the writer.
3. **Reads** overlap-scan the main index and the delta segment, merging the top-k results. 
4. When the delta segment fills, a background thread sequentially merges it into a new main segment and atomically swaps the segment list. 
*Result:* Zero COW during sweeps. Write amplification drops from 10MB to ~256 bytes (just the delta append).

```rust
// ---- Admission Control ----
// Qwen's backpressure, but charging fan-out cost (my R2 point survives:
// a delete costs F, not 1, or the churn-attacker wins).
pub struct AdmissionControl {
    pending_tasks: AtomicU64,   // P_total: unswept reconciliation tasks
    p_max: u64,                 // backpressure threshold
    sweep_rate: u64,            // R_sweep, composites/sec (for staleness calc)
    ingest_paused: AtomicBool,
    rejected_total: AtomicU64,  // monitoring
}

impl AdmissionControl {
    pub fn new(p_max: u64, sweep_rate: u64) -> Self {
        Self {
            pending_tasks: AtomicU64::new(0),
            p_max, sweep_rate,
            ingest_paused: AtomicBool::new(false),
            rejected_total: AtomicU64::new(0),
        }
    }

    /// Admit a write costing `fan_out` reconciliation tasks.
    /// CORRECTION to Qwen: cost is charged as fan_out, not 1.
    /// Otherwise a single insert (cost 1 to admit) can enqueue 10K sweep
    /// tasks, and pausing "the ingest thread" doesn't undo the 10K already
    /// queued. We must reject BEFORE the fan-out lands.
    pub fn try_admit(&self, fan_out: u64) -> Result<AdmitGuard, Rejected> {
        let cur = self.pending_tasks.load(Ordering::Acquire);
        if cur.saturating_add(fan_out) > self.p_max {
            self.ingest_paused.store(true, Ordering::Release);
            self.rejected_total.fetch_add(1, Ordering::Relaxed);
            return Err(Rejected { pending: cur, would_add: fan_out });
        }
        self.pending_tasks.fetch_add(fan_out, Ordering::AcqRel);
        Ok(AdmitGuard { ctl: self, tasks: fan_out })
    }

    fn release(&self, tasks: u64) {
        let prev = self.pending_tasks.fetch_sub(tasks, Ordering::AcqRel);
        if prev.saturating_sub(tasks) < self.p_max / 2 {
            self.ingest_paused.store(false, Ordering::Release); // hysteresis
        }
    }

    /// Worst-case staleness bound (seconds) given current queue.
    pub fn staleness_bound_secs(&self) -> f64 {
        self.pending_tasks.load(Ordering::Acquire) as f64 / self.sweep_rate as f64
    }
}

pub struct AdmitGuard<'a> { ctl: &'a AdmissionControl, tasks: u64 }
impl Drop for AdmitGuard<'_> {
    fn drop(&mut self) { self.ctl.release(self.tasks); }
}
pub struct Rejected { pub pending: u64, pub would_add: u64 }
```

## Cascading Delete State Machine

```rust
#[derive(Clone, Copy, PartialEq, Debug)]
#[repr(u8)]
pub enum AtomState {
    Live = 0,
    Tombstoned = 1,   // delete accepted, reads return conf=0
    Sweeping = 2,      // reconciliation in progress
    Reclaimed = 3,     // all composites cleaned, slot reusable
}

pub struct AtomEntry {
    state: AtomU8State,        // atomic state transitions
    fan_out: AtomicU32,        // remaining unswept composites
    indices: RwLock<Arc<[u32; K]>>, // sorted u32 index set (COW via Arc swap)
}

struct AtomU8State(AtomicU32);
impl AtomU8State {
    /// CAS-guarded transition; returns false on illegal edge (chaos safety).
    fn transition(&self, from: AtomState, to: AtomState) -> bool {
        self.0.compare_exchange(
            from as u32, to as u32,
            Ordering::AcqRel, Ordering::Acquire,
        ).is_ok()
    }
    fn load(&self) -> AtomState {
        match self.0.load(Ordering::Acquire) {
            0 => AtomState::Live, 1 => AtomState::Tombstoned,
            2 => AtomState::Sweeping, _ => AtomState::Reclaimed,
        }
    }
}

// Legal edges ONLY:
//   Live -> Tombstoned        (delete admitted)
//   Tombstoned -> Sweeping    (background sweep claims it)
//   Sweeping -> Reclaimed     (fan_out hit 0)
//   Sweeping -> Tombstoned    (sweep batch incomplete, requeue)
// Illegal (asserted in chaos test): any edge into Live, Reclaimed->*

pub struct DeleteEngine {
    atoms: Vec<AtomEntry>,
    reverse_idx: Vec<RwLock<Vec<u32>>>, // atom -> composite_ids (sharded)
    sweep_queue: RwLock<BinaryHeap<Reverse<SweepTask>>>,
    admission: Arc<AdmissionControl>,
}

#[derive(PartialEq, Eq)]
struct SweepTask { priority: u32, atom_id: u32, batch_lo: u32, batch_hi: u32 }
impl Ord for SweepTask {
    fn cmp(&self, o: &Self) -> std::cmp::Ordering { self.priority.cmp(&o.priority) }
}
impl PartialOrd for SweepTask {
    fn partial_cmp(&self, o: &Self) -> Option<std::cmp::Ordering> { Some(self.cmp(o)) }
}

impl DeleteEngine {
    pub fn delete(&self, atom_id: u32) -> Result<(), Rejected> {
        let entry = &self.atoms[atom_id as usize];
        let fan = entry.fan_out.load(Ordering::Acquire) as u64;
        let _guard = self.admission.try_admit(fan)?; // backpressure HERE
        // guard intentionally leaked into sweep accounting; released as
        // composites are reconciled (see sweep_batch).
        std::mem::forget(_guard);

        if !entry.state.transition(AtomState::Live, AtomState::Tombstoned) {
            // already deleted: idempotent, refund admission
            self.admission.release(fan);
            return Ok(());
        }
        // Enqueue sweep tasks batched by shard (amortize delta-append).
        self.enqueue_sweep(atom_id, fan as u32);
        Ok(())
    }

    fn enqueue_sweep(&self, atom_id: u32, fan: u32) {
        const BATCH: u32 = 1024;
        let mut q = self.sweep_queue.write().unwrap();
        let mut lo = 0;
        while lo < fan {
            let hi = (lo + BATCH).min(fan);
            // priority: smaller fan-out swept first (fairness vs fat deletes)
            q.push(Reverse(SweepTask { priority: fan, atom_id, batch_lo: lo, batch_hi: hi }));
            lo = hi;
        }
    }

    /// Background worker. Reconciles one batch into the LSM delta segment.
    pub fn sweep_batch(&self, delta: &DeltaSegment) -> bool {
        let task = { self.sweep_queue.write().unwrap().pop() };
        let Some(Reverse(t)) = task else { return false };
        let entry = &self.atoms[t.atom_id as usize];
        // Tombstoned -> Sweeping (idempotent if already sweeping)
        let _ = entry.state.transition(AtomState::Tombstoned, AtomState::Sweeping);

        let composites = self.reverse_idx[(t.atom_id as usize) % NUM_SHARDS].read().unwrap();
        let batch = &composites[t.batch_lo as usize..t.batch_hi as usize];
        for &cid in batch {
            // Append null-rebind to delta segment (NO COW of main index)
            delta.append_unbind(cid, t.atom_id);
        }
        let done = entry.fan_out.fetch_sub(batch.len() as u32, Ordering::AcqRel);
        // refund admission tokens for swept composites
        self.admission.release(batch.len() as u64);

        if done as usize == batch.len() {
            entry.state.transition(AtomState::Sweeping, AtomState::Reclaimed);
        } else {
            entry.state.transition(AtomState::Sweeping, AtomState::Tombstoned); // requeue rest
        }
        true
    }
}
```

## 1. Demolishing the 8GB Wall: Disk-Backed CSR

We do not need the reverse index in RAM. The reverse index (`atom_id -> Vec<composite_id>`) is **only** read by the background sweep worker. It is never on the hot read path. Therefore, we offload the 4GB edge list to disk using a **Compressed Sparse Row (CSR)** format, keeping only the 80MB offset array in RAM. 

Because sweep is a batched background job, the `pread` syscall latency is completely masked by the batch processing. Zero external crates, pure `std::os::unix::fs::FileExt`.

```rust
/// On-disk reverse index. 4GB edge list on disk, 80MB offset array in RAM.
pub struct DiskReverseIndex {
    file: File,
    /// offsets[i] is the byte offset where atom i's composite list starts.
    /// 10M atoms * 8 bytes = 80 MB.
    offsets: Vec<u64>, 
}

impl DiskReverseIndex {
    /// Read composite IDs for a deleted atom. Called only by background sweep.
    pub fn read_composites(&self, atom_id: u32) -> std::io::Result<Vec<u32>> {
        let start = self.offsets[atom_id as usize];
        let end = self.offsets[atom_id as usize + 1];
        let len = (end - start) as usize;
        if len == 0 { return Ok(Vec::new()); }

        self.file.read_exact_at(&mut buf, start)?;

            .map(|c| u32::from_ne_bytes(c.try_into().unwrap()))
            .collect())
    }
```
